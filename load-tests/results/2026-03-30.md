| Metric | Value | Target |

**Key finding:** At 100 RPS auth saturates because bcrypt (cost 12) limits login throughput to
~10 logins/s per CPU core. In production with multiple server replicas this scales linearly
with core count. 100 RPS auth is achievable with >=10 server instances. Refresh and logout
are fast (sub-5ms p99 — no bcrypt involved).

## Bugs Found and Fixed

### BUG-1 (HIGH): RATE_LIMIT_USER_RPM default too low for production

Default RATE_LIMIT_USER_RPM=100 means 100 requests-per-MINUTE per user/agent. At 100 RPS
this exhausts in 1 second, causing 65%+ failure rates. Default should be ~6000 (100 RPS x 60s).
Load tests ran with RATE_LIMIT_USER_RPM=20000 override to obtain valid measurements.
Tracking as backlog item to fix the default in env.ts.

### BUG-2 (MEDIUM): RBAC crash on agent run tokens — FIXED

apps/server/src/middleware/rbac.ts:29 — requirePermission() crashed with
"TypeError: roles is not iterable" when called with an agent run token (no roles claim).
Fixed: for (const role of (roles ?? [])) — agent tokens now correctly return 403 instead of 500.

### BUG-3 (LOW): atob not available in k6 v1.7.1 — FIXED

load-tests/websocket.js setup() used atob() (browser API, not in k6 runtime).
Fixed: replaced with encoding.b64decode(payload, "rawurl", "s") from k6/encoding.

---

## Design Finding: Auth throughput requires horizontal scaling

bcrypt cost 12 is correct for security. Single node limits login to ~10 RPS. Meeting the
100 RPS target requires >=10 API server replicas behind a load balancer. Document in ops
runbook. All non-auth endpoints (checkin, files, WebSocket) comfortably meet 100 RPS targets
on a single node with significant headroom.
